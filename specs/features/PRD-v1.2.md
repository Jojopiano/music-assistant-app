## 1. 問題陳述

目前系統無法讓老師與學生建立關係，導致所有現有功能（課表、打卡、進度追蹤、附件上傳）對新用戶完全無法啟動。這是影響系統成長的最關鍵缺口。

| 項目 | 說明 |
|------|------|
| **受影響用戶** | 所有新加入的老師與學生 |
| **目前替代方案** | 無（只能靠後台人工建立） |
| **造成的損失** | 新教室無法自助 onboard，成長完全依賴人工介入 |

## 2. 目標與成功指標

### 目標

- 讓老師能在 5 分鐘內邀請學生加入
- 讓學生能在 3 步驟內完成配對
- 讓老師能在課程頁面上傳並分享教學附件給學生

### 成功指標

| 指標 | 目標值 | 衡量方式 |
|------|--------|----------|
| 邀請連結點擊後配對完成率 | ≥ 70% | 事件追蹤 |
| 老師首次邀請時間 | ≤ 5 分鐘 | 操作時間追蹤 |
| 新用戶 onboard 需人工介入次數 | 趨近 0 | 客服紀錄 |
| 課程附件上傳後學生開啟率 | ≥ 60% | 事件追蹤 |

### 非目標（此版本不做）

- ❌ 學生主動搜尋老師
- ❌ 教室管理員審核流程
- ❌ 批量邀請（CSV 匯入）
- ❌ 學生上傳作業或錄音
- ❌ 獨立教材庫頁面

## 3. 用戶故事

### 老師端

**US-01**
> 身為老師，我想填寫個人資料（姓名、教室、專長、自我介紹），
> 這樣學生在確認配對時能清楚知道這是正確的老師。

**US-02**
> 身為老師，我想產生一個邀請代碼或連結，
> 這樣我可以透過 Line／簡訊傳給學生，不需要對方先有帳號。

**US-03**
> 身為老師，我想看到哪些學生「待確認」、哪些已配對成功，
> 這樣我能追蹤邀請狀態，不用一一詢問學生。

**US-04**
> 身為老師，我想能隨時解除與學生的配對關係，
> 這樣當學生停課時，我的學生列表能保持整潔。

**US-05**
> 身為老師，我想在每堂課程下上傳 PDF、照片等教材，
> 這樣學生課後可以隨時查看複習。

### 學生端

**US-06**
> 身為新學生，我想點開老師的連結後直接進入配對流程，
> 這樣我不需要自己去找老師，減少錯誤配對的風險。

**US-07**
> 身為學生，我想確認配對成功後才看到老師的課表資訊，
> 這樣我能確保自己連到正確的老師。

**US-08**
> 身為學生，我想在課程頁面查看老師上傳的教材並下載，
> 這樣我能在家練習時有譜可以看。

## 4. 功能規格

### 4.1 老師個人資料設定

**設定入口：** 整合進現有「設定」頁，新增「個人資料」區塊。

| 欄位 | 必填 | 類型 | 限制 |
|------|------|------|------|
| 姓名／暱稱 | ✅ 必填 | 文字 | 上限 20 字 |
| 所屬音樂教室 | 選填 | 文字（自由輸入） | 上限 50 字，可填多間 |
| 樂器專長 | 選填 | 文字（自由輸入） | 上限 30 字 |
| 自我介紹 | 選填 | 文字 | 上限 150 字 |
| 大頭照 | 選填 | 圖片 | 建議 1:1，上限 2MB |

**顯示規則：**
- 學生在配對確認頁看到：大頭照 ＋ 姓名 ＋ 教室 ＋ 樂器專長 ＋ 自我介紹
- 配對成立後，學生在「我的老師」頁可持續查看完整資料

**首次邀請引導：**
老師若尚未填寫個人資料就嘗試產生邀請代碼，系統顯示提示：
> 「建議先完善個人資料，讓學生更容易確認身份」

提供「前往設定」和「略過，直接產生代碼」兩個選項，不強制阻擋。

---

### 4.2 邀請代碼產生

```
規則：
- 6位英數混合代碼（例：A3F7K2）
- 有效期限：7 天（過期自動失效）
- 每位老師同時只有 1 組有效代碼
- 可手動重新產生（舊代碼立即失效）
- 同一代碼可被多位學生使用（一對多邀請）
```

### 4.3 邀請連結行為

```
格式：app.musichelper.com/invite/[CODE]

行為：
- 已登入        → 直接進入配對確認頁
- 未登入        → 導向註冊／登入頁 → 完成後自動跳回配對確認頁
- 代碼無效或過期 → 顯示錯誤提示，引導聯繫老師
```

### 4.4 配對確認頁（學生視角）

學生點開連結後看到：
- 老師大頭照
- 老師姓名
- 所屬教室
- 樂器專長
- 自我介紹
- 「確認配對」／「取消」按鈕

### 4.5 配對狀態管理

**老師端「我的學生」列表：**

| 欄位 | 說明 |
|------|------|
| 學生姓名 | 配對成立後顯示 |
| 配對狀態 | 已配對 ／ 邀請中（代碼未使用）|
| 加入時間 | 配對成立日期 |
| 操作 | 解除配對 |

**學生端「我的老師」頁面：**
- 顯示已配對老師的姓名、教室、專長、自我介紹
- 顯示「解除配對」選項

---

### 4.6 課程附件上傳

**上傳入口：** 每堂課程頁面下方「課程內容」區塊

**支援格式：**

| 類型 | 格式 |
|------|------|
| 文件 | PDF |
| 圖片 | JPEG、PNG、HEIC |
| 限制 | 單檔上限 10MB、每堂課上限 10 個檔案 |

**老師端操作流程：**
```
進入課程頁面
→ 點「新增內容」
→ 選擇：文字筆記 ／ 上傳檔案（可同時進行）
→ 上傳完成後預覽確認
→ 發布 → 學生收到通知
```

**學生端查看：**
- 只有該堂課的配對學生可見
- 可線上預覽 PDF 和圖片（不強迫下載）
- 可下載儲存至本機

**檔案命名規則：**
- 預設顯示原始檔名
- 老師可選擇重新命名（例：「第3週練習曲譜」）

## 5. 資料模型變更

```sql
-- 更新既有 Table：users
ALTER TABLE users ADD COLUMN display_name   VARCHAR(20);     -- 姓名／暱稱，必填
ALTER TABLE users ADD COLUMN studio_name    VARCHAR(50);     -- 音樂教室名稱，nullable
ALTER TABLE users ADD COLUMN instrument     VARCHAR(30);     -- 樂器專長，nullable
ALTER TABLE users ADD COLUMN bio            VARCHAR(150);    -- 自我介紹，nullable
ALTER TABLE users ADD COLUMN avatar_url     TEXT;            -- 大頭照儲存路徑，nullable

-- 新增 Table：invite_codes
CREATE TABLE invite_codes (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  teacher_id  UUID NOT NULL REFERENCES users(id),
  code        VARCHAR(6) UNIQUE NOT NULL,   -- 6碼唯一值
  expires_at  TIMESTAMPTZ NOT NULL,         -- created_at + 7天
  is_active   BOOLEAN DEFAULT TRUE,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 新增 Table：teacher_student_relationships
CREATE TABLE teacher_student_relationships (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  teacher_id   UUID NOT NULL REFERENCES users(id),
  student_id   UUID NOT NULL REFERENCES users(id),
  status       VARCHAR(20) DEFAULT 'pending', -- pending / active / dissolved
  created_at   TIMESTAMPTZ DEFAULT NOW(),
  activated_at TIMESTAMPTZ
);

-- 新增 Table：lesson_attachments
CREATE TABLE lesson_attachments (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  lesson_id    UUID NOT NULL REFERENCES lessons(id),
  uploader_id  UUID NOT NULL REFERENCES users(id),
  file_name    TEXT NOT NULL,               -- 顯示名稱（可自訂）
  file_url     TEXT NOT NULL,               -- Supabase Storage 路徑
  file_type    VARCHAR(10) NOT NULL,        -- pdf / jpeg / png / heic
  file_size_kb INTEGER,
  created_at   TIMESTAMPTZ DEFAULT NOW()
);
```

## 6. 邊界條件與錯誤處理

| 情境 | 系統行為 |
|------|----------|
| 代碼已過期 | 顯示「此邀請已過期，請向老師索取新代碼」 |
| 學生已與該老師配對 | 顯示「你已經是這位老師的學生了」 |
| 學生點連結但尚未註冊 | 保留代碼於 session，註冊完成後自動繼續配對流程 |
| 老師重新產生代碼 | 提示「舊代碼將立即失效，確定要重新產生？」 |
| 解除配對後歷史資料 | 保留歷史課程紀錄，僅解除關係，不刪除資料 |
| 上傳檔案超過 10MB | 即時提示「檔案過大，請壓縮後再上傳」 |
| 超過每堂課 10 個附件 | 提示「已達上限，請先刪除舊附件」 |
| 不支援的檔案格式 | 提示「僅支援 PDF、JPEG、PNG、HEIC 格式」 |
| 非授權學生存取附件 URL | 回傳 403，不回傳任何檔案內容 |

## 7. 技術考量

### 師生配對
- 邀請代碼需防碰撞：產生時檢查資料庫唯一性
- 連結需支援 deep link（若有 App 版本）
- Session 保留代碼需考慮 cookie 有效期（建議 30 分鐘）
- 解除配對為軟刪除（status 改為 dissolved），不物理刪除

### 附件權限控制
```
學生請求查看附件時，系統依序檢查：
1. 該學生是否與該老師配對？         → 否：拒絕（403）
2. 該課程是否屬於該師生關係？        → 否：拒絕（403）
3. 通過 → 回傳 Supabase signed URL（有效期 1 小時）
```

> ⚠️ 附件不能使用公開 URL，必須使用 Supabase Storage 的 signed URL 機制，確保只有授權學生能存取。Supabase 免費版已內建此功能。

### 推薦架構（最低成本）
- **資料庫 ＋ 認證 ＋ 檔案儲存：** Supabase（免費版支援 500MB DB ＋ 1GB Storage）
- **前端部署：** Vercel（免費版）
- **早期預估費用：** 幾乎 NT$0 / 月（500 用戶以內）
